# Topology

中文版：[topology.zh.md](./topology.zh.md)

A `Topology` is a cluster-scoped, read-only topology view generated by the
Controller. It has no `spec`; its `status` exposes performance domains, parent
relationships, switch members, and Node groups.

## Example

```yaml
apiVersion: unifabric.io/v1beta1
kind: Topology
metadata:
  name: scaleout
status:
  domains:
    - name: tier2-group1
      tier: 2
      members:
        - spine01
    - name: tier1-group1
      tier: 1
      parent: tier2-group1
      members:
        - leaf01
        - leaf02
  nodes:
    - domainPath:
        - tier2-group1
        - tier1-group1
      nodes:
        - node1
        - node2
```

This example means:

- `leaf01` and `leaf02` belong to `tier1-group1`.
- `spine01` belongs to their parent domain, `tier2-group1`.
- `node1` and `node2` share the same scale-out performance-domain path.

## Resource information

| Field | Value |
| --- | --- |
| API group | `unifabric.io` |
| API version | `v1beta1` |
| Kind | `Topology` |
| Scope | Cluster |
| Resource | `topologies` |
| Singular name | `topology` |
| Short name | `topo` |
| Status subresource | Yes |

## Fixed resource names

Only the following `metadata.name` values are accepted:

| Name | Description |
| --- | --- |
| `scaleout` | Scale-out network topology |
| `scaleup` | Scale-up network topology |
| `storage` | Storage network topology |

The Controller creates an object only after non-empty topology data exists for
that network. Therefore, `kubectl get topo` can be empty when there is no
topology input.

## Topology fields

| Field | Type | Description |
| --- | --- | --- |
| `apiVersion` | string | `unifabric.io/v1beta1` |
| `kind` | string | `Topology` |
| `metadata` | `metav1.ObjectMeta` | Standard Kubernetes metadata; the name must be one of the fixed values |
| `status` | `TopologyStatus` | Read-only aggregate maintained by the Controller |

`Topology` has no `spec`.

## TopologyStatus

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `domains` | `[]TopologyDomain` | No | Performance domains in the topology, unique by `name` |
| `nodes` | `[]TopologyNodeGroup` | No | Kubernetes Nodes grouped by complete domain path |

## TopologyDomain

| Field | Type | Required | Validation | Description |
| --- | --- | --- | --- | --- |
| `name` | string | Yes | Length `1`–`63` | Domain name and the value of Node topology labels and the Switch domain label |
| `tier` | int32 | Yes | Minimum `1` | Domain tier; `tier: 1` is nearest to a Node, and larger values are further upstream |
| `parent` | string | No | None | Name of the direct parent domain |
| `members` | `[]string` | No | Set with unique elements | Names of Switch CRs in this domain |

A root domain has no `parent`. Node-only mode has no real Switch CRs, so
`members` can be empty or omitted.

## TopologyNodeGroup

| Field | Type | Required | Validation | Description |
| --- | --- | --- | --- | --- |
| `nodes` | `[]string` | Yes | At least one item; set | Kubernetes Node names sharing the complete path |
| `domainPath` | `[]string` | Yes | At least one item | Domain names ordered from the highest tier to tier 1 |

For example, when a Node is in `tier1-group1` whose parent is
`tier2-group1`, its `domainPath` is:

```yaml
domainPath:
  - tier2-group1
  - tier1-group1
```

## Data source and ownership

`TopologyStatusController` aggregates status from labels on Kubernetes
objects:

- Tier labels on Nodes for the current network determine domain paths and Node
  groups.
- The `unifabric.io/domain` label on a Switch determines
  `status.domains[*].members`; `Switch.spec.role` selects the Topology to which
  it belongs.

`Topology.status` is not a direct input to topology discovery and does not
overwrite user configuration. Do not edit it manually.

In built-in discovery mode, deleting a `Topology` with the
`unifabric.io/auto-discovered-topology-labels` finalizer is an explicit reset.
The Controller clears the managed Node and Switch topology labels for that
network before allowing deletion. If discovery inputs remain, a later
reconciliation generates the labels and `Topology` again.

## Common commands

```bash
kubectl get topo
kubectl get topo scaleout -o yaml

kubectl get topo scaleout \
  -o jsonpath='{range .status.domains[*]}{.name}{"\t"}{.tier}{"\t"}{.parent}{"\n"}{end}'
```

For topology labels, aggregation rules, and reset behavior, see
[Topology CRD design](../design/topology-crd.md).
